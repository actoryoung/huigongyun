# 解析与检索接口约定

## 1. 目标

本项目将“输入解析”和“历史检索/相似匹配”拆成独立接口层，确保 Excel、PDF、Word、图片、DWG 等输入可以分别接入，后续的 OCR、版面分析、RAG 检索与相似物料匹配也可以独立替换。

## 2. 解析层接口边界

- `ProjectParser`：顶层入口，只负责把输入路径交给合适的格式解析器。
- `SourceParser`：单格式解析器，只负责一个文件族的识别与基础结构化，不承担 BOM、报价或校验逻辑。
- `ExcelSourceParser`：处理 Excel 工作簿，是当前唯一的正式实现。
- `PdfSourceParser` / `WordSourceParser` / `ImageSourceParser` / `DwgSourceParser`：当前为骨架实现，仅返回显式的占位项目文档。

## 3. 二级抽取接口

### 3.1 OCR 文本抽取

- `OCRTextExtractor`：面向扫描 PDF、图片和未来渲染后的图纸页面，输出 OCR 文字块和位置数据。
- `OCRBlock`：用于承载文本、置信度、bbox、页码等基础信息。
- `ExtractedDocument`：用于承载一组 OCR/Text 结果及其元数据。

### 3.2 文档文本抽取

- `DocumentTextExtractor`：面向 PDF、Word 等可直接读文本的文档，输出文本片段、表格片段和来源偏移。
- 该接口保留 OCR fallback 的可能性，但不直接做语义推断。

## 4. 检索与匹配接口边界

- `HistoricalCaseRetriever`：查询历史项目、柜体、报价或参考案例，返回排序后的候选命中。
- `CaseHit`：承载候选案例的标识、得分、摘要和可扩展载荷。
- `SimilarMaterialMatcher`：对材料名称、规格、品牌等做相似度计算，供归一与聚合阶段使用。

## 5. 当前实现状态

- 已实现：Excel 解析、柜体/BOM/报价/校验主流程。
- 已预留：PDF、Word、图片、DWG 的格式适配器骨架。
- 已预留：OCR 文本抽取、文档文本抽取、历史案例检索、相似物料匹配接口。

## 6. 后续接入顺序建议

1. OCR 与文档文本抽取接入开源实现。
2. 历史案例检索接入本地索引或 RAG。
3. 相似物料匹配接入 RapidFuzz 或 embedding 检索。
4. 在必要的高不确定场景再接商业大模型 API。

## 7. 说明

以上接口文档与当前代码一起构成可演进的 MVP 接口边界；后续新增实现时，只要遵守这些边界，就不需要改动主流水线签名。
