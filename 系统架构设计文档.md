# 汇工云系统架构设计文档

## 一、 架构概述
本项目旨在构建一个高可用、可扩展的低压电气成套项目智能报价清单生成系统。系统需满足非结构化文件（Excel/PDF/Word/DWG/图片）的高并发解析、智能 BOM 组装、基于规则与向量的相似检索，以及严格的人工审查修改追溯。 

系统由最小可行性产品（MVP）阶段向生产环境演进，整体采用**前后端分离 + 异步任务调度 + 微服务化 AI 层**的混合分层架构。

---

## 二、 核心业务分层

### 1. 表现层 (Frontend)
- **定位**：提供文件上传、系统解析状态追踪、BOM/报价清单可视化校验、以及人工比对回灌（Human-in-the-loop）的可视化操作界面。
- **技术选型**：
  - **MVP 阶段**：基于当前 `ExtJS` 或轻量模板渲染（如 `HTMX` / `Jinja2`）的快速表单。
  - **生产阶段**：`React` 或 `Vue.js` 单页应用（SPA）。涉及较多复杂的数据表格交互（类似 Excel 在线预览与编辑）。

### 2. 业务逻辑层 (Backend API)
- **定位**：系统的中枢大脑，负责处理鉴权、项目信息登记、任务编排、业务规则校验、数据查询及导出触发。
- **技术选型**：
  - **框架**：`FastAPI`（首选，对异步支持极好且输入验证友好）或 `Flask`。
  - **特性**：全部 RESTful 化，将耗时的解析流程（OCR、图纸读取）与主交互线程完全剥离。

### 3. 数据持久层 (Domain Database)
- **定位**：持久化项目元数据数据、价格表、柜体记录、BOM 记录及系统的历史操作修正。
- **技术选型**：
  - **MVP 阶段**：`SQLite`（开发快，零配置）+ 本地 JSON 文件。
  - **生产阶段**：`PostgreSQL`，擅长处理复杂的关联查询和 JSONB 数据，且支持后续结合 `PGVector` 扩展向量搜索能力。

---

## 三、 外围与支撑基础架构

除基本三层外，由于业务存在“重IO/CPU”、“AI检索”、“高要求数据溯源”等特殊要求，必须设计以下基础设施：

### 1. 异步任务队列与调度层 (Async Task Queue)
- **职责**：处理 PDF 解析、DWG 解析、长清单生成等耗时（秒级到分钟级）的任务。防止 Web 请求阻塞/超时。
- **选型**：`Redis` + `Celery` （或 `ARQ`）。通过任务队列发布解析任务，前端通过 Task ID 轮询或 WebSocket 接收进度更新。

### 2. 对象存储系统 (Object Storage)
- **职责**：存储原始用户上传文件（图纸、清单等），以及系统的输出产物（生成的导出 Excel 报告、中间缓存截图），实现同数据库的数据隔离。
- **选型**：`MinIO`（私有化部署的 S3 兼容级方案），支持分片传输和生命周期管理。

### 3. AI 推理与 OCR 隔离层 (AI Inference & OCR Worker)
- **职责**：专注处理深度图纸识别、多模态 OCR、大规模文本抽取任务。这部分通常需要显卡(GPU)算力或调用第三方大模型，环境依赖极为厚重（OpenCV, Poppler, CUDA 等）。
- **设计**：从主 Web 容器中**物理隔离**。设计为独立的 Python Microservice / RPC，或者直接桥接对应的云服务 API 服务接口。

### 4. 规则引擎与 RAG 向量检索层 (Rules & RAG Vector Stratum)
- **职责**：
  - **规则中心**：管理并热加载“不同柜型对应典型物料”、“品牌替换规则”等业务逻辑，实现配置变更不下线。
  - **检索中心**：实现历史项目的物料名称泛化匹配和相似案例检索。
- **选型**：`RapidFuzz`（MVP 期本地相似度计算） -> 演进到 `FAISS` / `Chroma` / `PGVector` 做海量数据的向量高维搜索。

### 5. 操作审计与血缘追踪存储 (Audit & Lineage)
- **职责**：实现数据的强溯源，记录“自动识别来源于哪张图纸/坐标”、“人工何时将物料A改成了物料B”、“价格更改原因”。
- **设计**：利用数据库的审计日志表（Audit Logs），或采用业务层的**事件溯源 (Event Sourcing)** 机制存储每一条 BOM 数据的演进版本 (Diff)。

### 6. 容器化部署架构 (Containerization)
- **职责**：抹平不同操作系统环境在 C++ 扩展、系统字体、渲染库配置上的巨大差异。
- **选型**：`Docker` & `Docker Compose`。项目需分别打出 `API-Server`, `Worker-Node` (带 AI 运行库), `DB` 和 `Redis` 等独立镜像进行部署。

---

## 四、 架构演进路线建议

- **Phase 1: MVP 验证闭环**
  FastAPI (或现有轻量演示壳) + SQLite + 本地存储 + 本地同步解析/RapidFuzz。目标跑通业务流转逻辑。
- **Phase 2: 生产可用化**
  引入 Redis/Celery 解除请求超时；使用 PostgreSQL + MinIO 正规化数据和文件流；上架容器化编排。
- **Phase 3: 深水区 AI 支持拓展**
  独立拓展并接入图纸 OCR 工作流服务器；引入向量数据库强化历史数据的复用组装效率。
